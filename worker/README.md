## Runtime Modes

The Worker uses the same code locally and in Render. Behavior is controlled by environment variables.

```env
WORKER_POLL_ENABLED=false
WORKER_MOCK_MODE=true
```

Common modes:

| Mode | WORKER_POLL_ENABLED | WORKER_MOCK_MODE | Purpose |
|---|---:|---:|---|
| Safe standby | false | true/false | Start Worker without claiming jobs |
| Local mock | true | true | Claim jobs without RunPod cost |
| Local real test | true | false | Claim jobs and call RunPod |
| Render production | true | false | Production job processing |

`WORKER_POLL_ENABLED` defaults to `false`. A local Worker will not claim production jobs unless this is explicitly set to `true`.

## Required Environment Variables

Do not commit real `.env` files.

Minimum production variables:

```env
NODE_ENV=production
SUPABASE_URL=
SUPABASE_SERVICE_ROLE_KEY=
RUNPOD_API_KEY=
INTERNAL_API_SECRET=
WORKER_CALLBACK_URL=https://www.ymistory.com/api/internal/worker-callback
WORKER_POLL_ENABLED=true
WORKER_MOCK_MODE=false
WORKER_ID=render-ymi-worker-1
WORKER_JOB_TYPES=preview,final
WORKER_LEASE_SECONDS=1800
WORKER_LEASE_RENEW_INTERVAL_MS=120000
HEALTHCHECKS_URL=
```

Existing RunPod / workflow timeout / compression variables should be copied from the current working local Worker profile into Render.

## Supabase SQL

Before enabling Render polling, run:

```text
Template_folder/sql_worker_claim_lease.sql
```

This adds:

- `jobs.claimed_by`
- `jobs.claimed_at`
- `jobs.lease_expires_at`
- `jobs.claim_attempts`
- `claim_next_job(p_worker_id, p_job_types, p_lease_seconds)`
- compatible no-argument `claim_next_job()`
- `renew_job_lease(p_job_id, p_worker_id, p_lease_seconds)`

## Render Deployment

Recommended Render settings:

- Service type: Background Worker
- Region: US East
- Root directory: `worker`
- Build: Dockerfile
- First deploy:
  - `WORKER_POLL_ENABLED=false`
  - `WORKER_MOCK_MODE=false`
- After logs and health look correct:
  - switch `WORKER_POLL_ENABLED=true`

## Health

The Worker starts a local health endpoint:

```text
http://127.0.0.1:8787/health
```

The JSON includes:

- worker id
- poll enabled
- mock mode
- job types
- current jobs
- last claim poll
- last Supabase success
- last error

Healthchecks.io should only be configured on the environment that is actually responsible for production processing.

## Production Cutover Checklist

1. Run the Supabase SQL.
2. Deploy Render with `WORKER_POLL_ENABLED=false`.
3. Confirm startup logs show the expected env mode.
4. Stop local production Worker or set local `.env` to `WORKER_POLL_ENABLED=false`.
5. Switch Render to `WORKER_POLL_ENABLED=true`.
6. Validate one preview job.
7. Validate one full final job:
   - pages enter `final_job_pages.status='pending_review'`
   - `final_jobs.status='review_pending'`
   - Admin release succeeds
   - PDF is generated by Next.js `releaseFinalJob()`
   - final delivery email is logged in `email_events`

## Legacy Notes

`/api/internal/worker-callback` and the worker-side `buildPdf()` branch are legacy/non-review fallback paths. Current production final delivery goes through Admin review and Next.js `releaseFinalJob()`.
